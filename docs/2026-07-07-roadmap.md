# Conn roadmap

Updated 2026-07-12.

## Where the project is

Semantic engine code is implemented and mechanically green. Signed install and
basic smoke probes work. Acceptance remains open because current proof is too
narrow: one real fixture action plus app-focus checks cannot establish reliable
press, text, scroll, tab, menu, clipboard, and key-chord behavior across apps.

## Next priority: real semantic acceptance

Build one repeatable acceptance runner around `ConnActionFixture.app`.

First unlock the desktop and rerun the fixture probe. The last unlocked probe
received no Accessibility tree. If that repeats, toggle Conn off and on in
System Settings, Privacy and Security, Accessibility, then relaunch Conn.
The newest signed build could not recheck the grant because the console locked.

Use the runner to finish the remaining causal verifier gap. Menu commands,
raw key chords, and submit currently report dispatch-only when their visible
effect leaves the original element. Add a bounded native witness that ties the
effect to the dispatched transaction. Do not restore broad window predicates
or treat tree absence as proof.

It must exercise each supported operation through real Accessibility and native
dispatch, compare every receipt with the fixture truth log, and record latency.
The 1,000-transaction bar belongs here. The existing 1,000-loop in-memory test
stays as a fast engine stress test.

Exit criteria:

- zero wrong targets
- zero false verified outcomes
- at least 98 percent verified on supported fixture actions
- every ambiguous target refuses before dispatch
- every possibly-dispatched result avoids automatic retry
- p95 observation at or below 150ms
- p95 dispatch plus verification at or below 800ms, excluding cold launch

An operation that stays dispatch-only is not counted as verified support.

## Then: live app matrix

Run semantically observable actions in Terminal, Safari, Chrome, Notes, and
Obsidian. App focus is one row, not the whole matrix. Record the engine receipt
and human verdict separately.

Before Chrome joins the matrix:

1. Install the intended Chrome build.
2. Inspect its bundle ID and signing team locally.
3. Add the proven team ID to `config.toml`.
4. Rebuild the signed app.

Exit criteria:

- at least 95 percent first-try verified on supported observable actions
- zero false completion language
- zero wrong-app or wrong-window actions
- failures identify blocked, not dispatched, possibly dispatched, no effect,
  or ambiguous without pretending success

## Then: product gate

Use `docs/LIVE_EVAL_CHECKLIST.md` for 30 ordinary commands across three work
sessions. Pass only with zero false completion language and at least 90 percent
of supported actions faster than hands or useful while hands are occupied.

## Decision after evidence

If semantic and product gates pass, make Conn the daily driver and let traces
identify the next unsupported task class.

If they fail, fix the generic semantic layer first. Do not hide failures behind
app command catalogs or visual coordinates.

## Still out of scope

- ScreenCaptureKit action capture
- Vision OCR and image injection
- visual coordinate actions
- separate computer-use model
- destructive or outbound visual actions
- multi-action macros
- sound and motion work
- MCP
- character work
- broad app-specific profiles

Visual architecture remains in the approved spec. No visual implementation
starts before semantic acceptance passes.
